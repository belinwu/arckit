# ArcKit Command Prompt Optimiser

## Overview

An autonomous experimentation loop — inspired by [karpathy/autoresearch](https://github.com/karpathy/autoresearch) — that iteratively improves ArcKit command prompts by running them, evaluating output quality, tweaking the prompt, and keeping or discarding the change based on whether the score improved.

The deliverable is a `program.md` file that instructs Claude Code to run this loop hands-free. No custom tooling. The LLM is the researcher.

## Core Principle

Autoresearch optimises `train.py` against `val_bpb`. This optimises a command `.md` file against a two-layer quality score. Same mechanics: branch-per-run, one change per iteration, keep/discard on delta, loop forever.

## Setup Phase

When starting a run, the program instructs Claude to:

1. Accept a target command name from the user (e.g. `requirements`).
2. Create branch `autoresearch/<command>-<date>` (e.g. `autoresearch/requirements-mar20`).
3. Read three files for full context:
   - `arckit-claude/commands/<command>.md` — the prompt to optimise
   - `arckit-claude/templates/<command>-template.md` — the expected output structure (read-only)
   - `arckit-claude/references/quality-checklist.md` — general quality criteria (read-only)
4. Create a disposable scratch project: `mkdir -p scratch/projects/001-test-project/` with a minimal `README.md`.
5. Establish baseline: run the command as-is against the scratch project, score the output, log the result as the first row in `results.tsv`.

## Evaluation — Two Layers

### Layer 1: Structural Checks (pass/fail gate)

Verified by reading the generated artifact and checking for:

- Document Control table present with all required fields (Document ID, Document Type, Project, Classification, Status, Version, Created Date, Last Modified, Review Cycle, Next Review Date, Owner, Reviewed By, Approved By, Distribution)
- Document ID follows `ARC-NNN-TYPE-vX.Y` pattern
- Revision History table present with required columns (Version, Date, Author, Changes, Approved By, Approval Date)
- Standard footer present (Generated by, Generated on, ArcKit Version, Project, Model)
- All major sections from the template file are present in the output
- File written to a path under `scratch/projects/`
- Domain-specific ID formats used correctly where applicable (BR-xxx, FR-xxx, NFR-xxx, INT-xxx, DR-xxx for requirements; RISK-xxx for risk registers; etc.)

A structural failure means the iteration is scored `FAIL 0.0` and discarded immediately.

### Layer 2: LLM-as-Judge (qualitative score, 1.0–10.0)

After passing structural checks, Claude evaluates the generated artifact on five dimensions, each scored 1–10:

| Dimension | What it measures |
|-----------|-----------------|
| **Completeness** | Are all sections substantively filled? No empty placeholders, no "TBD" without justification |
| **Specificity** | Does content reference the actual project context, not generic boilerplate? |
| **Traceability** | Are cross-references between artifacts present and correctly formatted? |
| **Actionability** | Could a vendor, architect, or review board use this document as-is? |
| **Clarity** | Is the writing clear, well-structured, and free of contradictions? |

The combined score is the arithmetic mean of the five dimensions, rounded to one decimal place.

## The Experiment Loop

```
LOOP FOREVER:

1. Read the current command prompt and review results.tsv history
2. Identify ONE specific improvement to the prompt
   - Draw ideas from: low-scoring dimensions, structural failures,
     template gaps, quality checklist criteria, or prompt engineering
     best practices
3. Edit arckit-claude/commands/<command>.md with the change
4. Git commit with a short description of what changed
5. Clean the scratch project (delete previously generated artifacts)
6. Run the command against the scratch project
   - The command will read the template, create the project structure,
     and write the artifact
7. Score the output:
   a. Run structural checks — if FAIL, skip LLM scoring, score = 0.0
   b. If PASS, run LLM-as-judge across five dimensions
   c. Compute combined score (mean of five dimensions)
8. Log to results.tsv:
   commit | structural | score | status | description
9. Compare to previous best score:
   - If improved (higher score) → KEEP (advance the branch)
   - If equal or worse → DISCARD (git reset --hard to previous commit)
10. Go to step 1
```

## Results TSV

Tab-separated, NOT comma-separated. Header row followed by one row per experiment.

```
commit	structural	score	status	description
a1b2c3d	PASS	6.8	keep	baseline
b2c3d4e	PASS	7.2	keep	added explicit instruction to fill all NFR subcategories
c3d4e5f	FAIL	0.0	discard	removed Document Control instruction (broke structure)
d4e5f6g	PASS	7.1	discard	reordered sections (no improvement)
e5f6g7h	PASS	7.6	keep	added example requirement IDs in prompt
```

## Constraints

These mirror autoresearch's rules:

- **Only modify the command `.md` file.** The template, quality checklist, evaluation criteria, and scratch project scaffolding are read-only — like autoresearch's `prepare.py`.
- **One change per iteration.** Isolate variables. If you change two things and the score improves, you don't know which helped.
- **Never stop.** Do not pause to ask the human. Keep iterating until manually interrupted. If you run out of ideas, re-read the template, the quality checklist, and the results history for new angles.
- **Simplicity criterion.** A marginal score improvement that adds significant prompt complexity is not worth it. Conversely, simplifying the prompt while maintaining the score is a win — keep it.
- **No new dependencies.** Don't introduce MCP servers, external tools, or scripts that the command doesn't already use.
- **Log everything.** Every iteration gets a row in `results.tsv`, whether kept or discarded. The history is the research record.

## What Is Read-Only

| File | Why |
|------|-----|
| `arckit-claude/templates/<command>-template.md` | The template defines expected output structure — changing it would move the goalposts |
| `arckit-claude/references/quality-checklist.md` | The evaluation standard must be stable for scores to be comparable |
| Evaluation rubric (above) | Same reason — fixed metric like `val_bpb` |
| Scratch project scaffolding | Controlled input so the only variable is the prompt |

## What Is NOT In Scope

- No custom Python/bash scripts for evaluation
- No CI integration
- No multi-command sweeps in a single run (one command per branch)
- No template modifications
- No changes to the plugin structure or agent system

## File Location

The `program.md` file will live at `scripts/autoresearch/program.md`. The results TSV and scratch project are ephemeral — created on the experiment branch, not merged to main.

## Usage

```bash
# Start a run
# Human tells Claude: "optimise the requirements command"
# Claude reads program.md, follows the setup, and enters the loop
```

The human's only interactions are: starting the run, and optionally stopping it.
